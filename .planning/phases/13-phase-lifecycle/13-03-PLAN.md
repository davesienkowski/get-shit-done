<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User args -> phaseComplete | Phase number from CLI |
| phaseComplete -> ROADMAP/REQUIREMENTS/STATE | Multi-file atomic writes |
| phasesClear -> filesystem | Bulk directory deletion |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-09 | Tampering | phaseComplete multi-file writes | mitigate | Each file has its own atomic lock cycle (ROADMAP lock, then REQUIREMENTS write, then STATE lock); partial failure leaves at most one file inconsistent |
| T-13-10 | Tampering | phasesClear bulk delete | mitigate | Requires --confirm flag; skips 999.x backlog dirs; only operates within .planning/phases/ |
| T-13-11 | Denial of Service | phasesArchive rename across volumes | accept | rename() may fail across filesystem boundaries; user should ensure .planning/ is on single volume |
| T-13-12 | Repudiation | phaseComplete silent warnings | accept | Warnings are returned in result object; caller (workflow/agent) decides whether to surface them |
</threat_model>
