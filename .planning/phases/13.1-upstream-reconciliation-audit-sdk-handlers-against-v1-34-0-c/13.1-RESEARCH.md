# Phase 13.1: Upstream Reconciliation - Research

**Researched:** 2026-04-08
**Domain:** SDK/CJS behavioral parity -- locking, atomicity, config keys, security guards
**Confidence:** HIGH

## Summary

Line-by-line comparison of SDK handlers (Phases 9-12) against the current v1.34.0 CJS codebase reveals **7 concrete divergences** across three categories: locking behavior (3 issues), config atomicity/completeness (3 issues), and security guards (1 issue). None are catastrophic -- the SDK implementations are functionally correct -- but they diverge from CJS behaviors in ways that could cause subtle issues in concurrent agent scenarios or fail golden tests after upstream merges.

The most impactful divergences are: (1) SDK `acquireStateLock` uses `setTimeout` instead of CJS's `Atomics.wait` for retry delays, (2) SDK lacks the `process.on('exit')` stale-lock cleanup that CJS added in PR #1916, and (3) SDK `atomicWriteConfig` uses a non-PID-qualified temp file name and lacks a rename-failure fallback. The config key allowlist is also 2 keys behind CJS.

**Primary recommendation:** Fix all 7 divergences in SDK code, re-run golden tests, and re-verify Phase 9-12 must-haves against updated codebase.

## Divergence Inventory

### D1: acquireStateLock retry uses setTimeout instead of Atomics.wait [VERIFIED: source comparison]

**SDK (state-mutation.ts:135):**
```typescript
await new Promise<void>(r => setTimeout(r, retryDelay + Math.floor(Math.random() * 50)));
```

**CJS (state.cjs:886):**
```javascript
Atomics.wait(new Int32Array(new SharedArrayBuffer(4)), 0, 0, retryDelay + jitter);
```

**Impact:** `setTimeout` is async and yields the event loop. `Atomics.wait` is synchronous and blocks the thread. In CJS, `acquireStateLock` is synchronous (uses `openSync`, `writeSync`, `closeSync`, `statSync`, `unlinkSync`). The SDK made the function async, which is a valid design choice, but the retry delay mechanism should still block for the delay period. Since the SDK is already async, `setTimeout` via Promise is actually fine -- the behavior matches. **However**, the CJS busy-wait removal PR #1944 specifically chose `Atomics.wait` to avoid `while(Date.now() < target)` spin loops. The SDK's async `setTimeout` achieves the same goal through a different mechanism.

**Verdict:** NOT A BUG -- SDK's async `setTimeout` is functionally equivalent to CJS's sync `Atomics.wait` since the SDK function is already async. No fix needed.

### D2: SDK lacks process.on('exit') lock cleanup [VERIFIED: source comparison]

**CJS (state.cjs:16-23):**
```javascript
const _heldStateLocks = new Set();
process.on('exit', () => {
  for (const lockPath of _heldStateLocks) {
    try { require('fs').unlinkSync(lockPath); } catch { /* already gone */ }
  }
});
```

The CJS adds the lock path to `_heldStateLocks` on acquire (line 869) and removes it on release (line 896). This ensures that if `process.exit(1)` is called inside a locked region (e.g., CJS's `error()` function calls `process.exit(1)`), the lock file is still cleaned up.

**SDK (state-mutation.ts):** No equivalent mechanism. The SDK uses async `try/finally` which should handle normal error paths, but does NOT handle `process.exit()` scenarios.

**CJS (core.cjs:31-36):** Same pattern exists for `_heldPlanningLocks` in `withPlanningLock`.

**Impact:** If the SDK process exits abnormally while holding a lock (e.g., process killed, uncaught exception reaching the event loop), the lock file remains on disk until the 10-second stale cleanup threshold passes. In practice, the SDK is a library (not a CLI that calls `process.exit()`), so this is LOW risk but still a behavioral divergence.

**Verdict:** BUG -- should add `process.on('exit')` cleanup to match CJS robustness. Even in library mode, unexpected termination can leave stale locks.

### D3: SDK acquireStateLock non-EEXIST error handling differs [VERIFIED: source comparison]

**CJS (state.cjs:889):**
```javascript
return lockPath; // non-EEXIST error -- proceed without lock
```

**SDK (state-mutation.ts:137):**
```typescript
throw err;
```

**Impact:** CJS silently proceeds without a lock on non-EEXIST errors (e.g., permission denied). SDK throws, which could crash the calling code. The CJS approach is more resilient in degraded environments but hides potential real errors.

**Verdict:** INTENTIONAL DIVERGENCE -- SDK's stricter error handling is arguably better. However, for reconciliation parity, should match CJS behavior (return lockPath on non-EEXIST) or document the intentional difference. Recommend matching CJS behavior since the locking is a best-effort mechanism.

### D4: SDK atomicWriteConfig temp file name lacks PID qualifier [VERIFIED: source comparison]

**SDK (config-mutation.ts:35):**
```typescript
const tmpPath = configPath + '.tmp';
```

**CJS (core.cjs:1530):**
```javascript
const tmpPath = filePath + '.tmp.' + process.pid;
```

**Impact:** Without PID qualification, concurrent SDK processes writing config could collide on the same `.tmp` file. The PID suffix prevents this. This matters in parallel agent scenarios (parallelization > 1).

**Verdict:** BUG -- should use PID-qualified temp path to match CJS.

### D5: SDK atomicWriteConfig lacks rename-failure fallback [VERIFIED: source comparison]

**SDK (config-mutation.ts:36-37):**
```typescript
await writeFile(tmpPath, ...);
await rename(tmpPath, configPath);
```

**CJS (core.cjs:1529-1538):**
```javascript
try {
  fs.writeFileSync(tmpPath, content, encoding);
  fs.renameSync(tmpPath, filePath);
} catch (renameErr) {
  try { fs.unlinkSync(tmpPath); } catch { /* already gone */ }
  fs.writeFileSync(filePath, content, encoding);
}
```

**Impact:** CJS falls back to direct write if rename fails (cross-device, permission issue), cleaning up the temp file. SDK would leave a stale `.tmp` file and throw an error. On Windows, rename can fail more often (file locking).

**Verdict:** BUG -- should add rename-failure fallback and temp cleanup to match CJS resilience.

### D6: SDK configSet lacks withPlanningLock [VERIFIED: source comparison]

**CJS (config.cjs:296):**
```javascript
function setConfigValue(cwd, keyPath, parsedValue) {
  return withPlanningLock(cwd, () => { ... });
}
```

**SDK (config-mutation.ts:172):**
```typescript
export const configSet: QueryHandler = async (args, projectDir) => {
  // No lock around read-modify-write cycle
  const raw = await readFile(paths.config, 'utf-8');
  config = JSON.parse(raw);
  setConfigValue(config, keyPath, parsedValue);
  await atomicWriteConfig(paths.config, config);
};
```

**Impact:** CJS wraps config writes in `withPlanningLock` to prevent concurrent agents from clobbering each other's changes. The SDK has no such protection, creating a TOCTOU race: Agent A reads config, Agent B reads config, Agent A writes, Agent B overwrites with stale data.

**Verdict:** BUG -- SDK config mutations need lock protection. Options: reuse `acquireStateLock` on the config path, or implement a general-purpose file lock.

### D7: SDK VALID_CONFIG_KEYS missing 2 keys [VERIFIED: source comparison]

**CJS (config.cjs:34-35):**
```javascript
'features.global_learnings',
'learnings.max_inject',
```

**SDK (config-mutation.ts:48-72):** Neither key is present in the `VALID_CONFIG_KEYS` Set.

Note: `features.global_learnings` would be accepted by the SDK's dynamic `features.*` regex pattern, but `learnings.max_inject` would be rejected as an unknown key.

**Impact:** SDK would reject `config-set learnings.max_inject <value>` with a "Unknown config key" error, while CJS accepts it. The `features.global_learnings` key works via the dynamic pattern but should still be in the explicit set for discoverability.

**Verdict:** BUG -- add both keys to SDK's VALID_CONFIG_KEYS.

### D8: SDK configSet lacks context value validation [VERIFIED: source comparison]

**CJS (config.cjs:357-359):**
```javascript
const VALID_CONTEXT_VALUES = ['dev', 'research', 'review'];
if (keyPath === 'context' && !VALID_CONTEXT_VALUES.includes(String(parsedValue))) {
  error(`Invalid context value '${value}'. Valid values: ${VALID_CONTEXT_VALUES.join(', ')}`);
}
```

**SDK:** No equivalent validation for the `context` key.

**Impact:** SDK accepts any value for `context` key, while CJS restricts to `dev|research|review`.

**Verdict:** BUG -- should add context value validation to match CJS.

### D9: SDK lacks CONFIG_KEY_SUGGESTIONS for typo correction [VERIFIED: source comparison]

**CJS (config.cjs:57-67):**
```javascript
const CONFIG_KEY_SUGGESTIONS = {
  'workflow.nyquist_validation_enabled': 'workflow.nyquist_validation',
  'agents.nyquist_validation_enabled': 'workflow.nyquist_validation',
  // ... 8 more suggestions
};
```

**SDK (config-mutation.ts:95-113):** Uses longest-common-prefix matching instead of explicit suggestion map.

**Impact:** CJS has curated suggestions for known typos (e.g., `workflow.codereview` -> `workflow.code_review`). SDK's LCP approach may not suggest the same corrections. For example, `workflow.codereview` would match `workflow.code_review` by LCP, but `agents.nyquist_validation_enabled` would not match `workflow.nyquist_validation` by LCP.

**Verdict:** MINOR DIVERGENCE -- SDK approach is more general but less precise. Consider adding the explicit suggestion map as a first-pass before LCP fallback.

### D10: SDK lacks validateFieldName security guard [VERIFIED: source comparison]

**CJS (state.cjs:189-191):**
```javascript
const { validateFieldName } = require('./security.cjs');
const fieldCheck = validateFieldName(field);
if (!fieldCheck.valid) { error(`state update: ${fieldCheck.error}`); }
```

The CJS validates field names with regex `^[A-Za-z][A-Za-z0-9 _.\-/]{0,60}$` to prevent regex injection via crafted field names passed to `stateUpdate` and `statePatch`.

**SDK (state-mutation.ts):** Uses `escapeRegex(fieldName)` in `stateReplaceField` which escapes special regex characters, providing equivalent protection against regex injection. However, it does not reject overly long or malformed field names.

**Impact:** The SDK's `escapeRegex` approach prevents regex injection (the primary threat), making this functionally equivalent from a security standpoint. The length limit and character allowlist in CJS are defense-in-depth. LOW risk since the field name comes from agents, not untrusted user input.

**Verdict:** MINOR DIVERGENCE -- `escapeRegex` provides equivalent regex injection protection. The allowlist/length check is defense-in-depth that could be added but is not a security gap.

### D11: CJS buildNewProjectConfig loads ~/.gsd/defaults.json -- SDK does not [VERIFIED: source comparison]

**CJS (config.cjs:103-120):** Loads global defaults from `~/.gsd/defaults.json` and does three-level merge: hardcoded <- userDefaults <- userChoices.

**SDK (config-mutation.ts:259-360):** Only does two-level merge: hardcoded <- userChoices. Does not load `~/.gsd/defaults.json`.

**Impact:** Users who set global defaults via `~/.gsd/defaults.json` would not get those defaults applied when the SDK creates new projects. However, `configNewProject` is rarely called (only during `/gsd-new-project`).

**Verdict:** FEATURE GAP -- should be added for parity, but LOW priority since new project creation typically goes through gsd-tools.cjs.

### D12: CJS configNewProject does not include `features: {}` in defaults [VERIFIED: source comparison]

**CJS (config.cjs:122-187):** The hardcoded defaults do NOT include `features: {}`.

**SDK (config-mutation.ts:326):** Includes `features: {}` in the default config.

**Impact:** SDK-created configs have an empty `features` key that CJS-created configs lack. This is harmless since both handle missing `features` key gracefully.

**Verdict:** NOT A BUG -- SDK is slightly more complete. CJS will work fine with the extra key.

## Architecture Patterns

### Locking Pattern (match CJS)

The SDK should add process-exit cleanup alongside its existing try/finally pattern:

```typescript
// Module-level stale lock cleanup
const _heldLocks = new Set<string>();
process.on('exit', () => {
  const { unlinkSync } = require('node:fs');
  for (const lockPath of _heldLocks) {
    try { unlinkSync(lockPath); } catch { /* already gone */ }
  }
});

// In acquireStateLock:
_heldLocks.add(lockPath);
return lockPath;

// In releaseStateLock:
_heldLocks.delete(lockPath);
try { await unlink(lockPath); } catch { /* already gone */ }
```

### Atomic Write Pattern (match CJS)

```typescript
async function atomicWriteConfig(configPath: string, config: Record<string, unknown>): Promise<void> {
  const tmpPath = configPath + '.tmp.' + process.pid;
  const content = JSON.stringify(config, null, 2) + '\n';
  try {
    await writeFile(tmpPath, content, 'utf-8');
    await rename(tmpPath, configPath);
  } catch {
    try { await unlink(tmpPath); } catch { /* already gone */ }
    await writeFile(configPath, content, 'utf-8');
  }
}
```

### Config Lock Pattern

```typescript
export const configSet: QueryHandler = async (args, projectDir) => {
  const paths = planningPaths(projectDir);
  const lockPath = await acquireStateLock(paths.config);
  try {
    // read-modify-write inside lock
    let config: Record<string, unknown> = {};
    try { config = JSON.parse(await readFile(paths.config, 'utf-8')); } catch { /* empty */ }
    setConfigValue(config, keyPath, parsedValue);
    await atomicWriteConfig(paths.config, config);
  } finally {
    await releaseStateLock(lockPath);
  }
};
```

## Common Pitfalls

### Pitfall 1: Lock File Left Behind After Tests
**What goes wrong:** Test creates lock file, test assertion fails, lock file not cleaned up, subsequent tests fail with stale lock.
**How to avoid:** Always use try/finally in tests. Add afterEach cleanup for any `.lock` files in tmpDir.

### Pitfall 2: Breaking Golden Tests by Changing Output Shape
**What goes wrong:** Adding a field to config-set output (e.g., `previousValue`) makes golden test fail because SDK output no longer matches CJS shape.
**How to avoid:** Golden tests compare specific fields, not entire objects. Ensure fixes only change internal behavior, not output shape.

### Pitfall 3: Windows rename() Failures
**What goes wrong:** On Windows, `rename()` (and `fs.renameSync`) can fail if the target file is open by another process. The CJS fallback handles this; SDK without fallback would throw.
**How to avoid:** Always implement the rename-failure fallback in atomic writes.
