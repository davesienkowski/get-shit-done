# Roadmap: GSD Enhancement Initiative — PBR Backport

## Overview

This milestone systematically backports proven innovations from PBR v2.22.0 into the GSD framework. The delivery order is dependency-driven: shared reference docs and config schema come first because every subsequent feature consumes them, followed by gates and execution hardening, then new user-facing commands, then knowledge uplift, and finally the intel system as a standalone capped phase. Each phase delivers a coherent, verifiable capability that unblocks the next.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 1: Shared Reference Foundation** - Establish the shared reference docs and agent contracts that all subsequent phases depend on
- [ ] **Phase 2: Configuration & Model Profiles** - Extend config.json schema with model profiles, execution context, and feature flags
- [x] **Phase 3: Gates & Execution Hardening** - Formalize the gate system, wave execution, and phase manifest infrastructure (completed 2026-04-04)
- [x] **Phase 4: New User Skills** - Deliver explore, continue, undo, and import commands (completed 2026-04-04)
- [ ] **Phase 5: Knowledge & Quality Uplift** - Add thinking model docs, learnings system, scan command, and status line integration
- [x] **Phase 6: Intel System** - Build the queryable codebase intelligence system as a bounded standalone phase (completed 2026-04-04)

## Phase Details

### Phase 1: Shared Reference Foundation
**Goal**: All GSD agents reference a consistent set of shared behavioral docs that prevent anti-patterns, define agent handoff contracts, and standardize gate prompts and revision loops
**Depends on**: Nothing (first phase)
**Requirements**: FNDTN-01, FNDTN-02, FNDTN-03, FNDTN-04, FNDTN-05, FNDTN-06, FNDTN-07, FNDTN-08
**Success Criteria** (what must be TRUE):
  1. Six shared reference docs exist in `get-shit-done/references/` covering anti-patterns, context budget, gate prompts, revision loop, domain probes, and agent contracts
  2. Agent contracts reference doc lists every GSD agent with its exact completion marker string — any new agent added in later phases must register here first
  3. Existing GSD workflows (plan-phase, execute-phase, discuss-phase) each contain at least one @-reference to the new shared docs
  4. No PBR-ism naming (pbr, pbr-tools, pbr:, CONTEXT.md) appears in any new file
  5. Each new reference doc is 150 lines or fewer (context budget guard)
**Plans**: 2 plans
Plans:
- [x] 01-01-PLAN.md — Create 6 shared reference docs (anti-patterns, context-budget, gate-prompts, revision-loop, domain-probes, agent-contracts)
- [x] 01-02-PLAN.md — Update 3 workflows with @-references and verify agent markers

### Phase 2: Configuration & Model Profiles
**Goal**: GSD's config.json schema is extended with model profiles, per-agent overrides, execution context profiles, and feature flags — all backwards-compatible with existing project configs
**Depends on**: Phase 1
**Requirements**: CONFIG-01, CONFIG-02, CONFIG-03, CONFIG-04, CONFIG-05, CONFIG-06, CONFIG-07, CONFIG-08
**Success Criteria** (what must be TRUE):
  1. Four model profile presets (quality/balanced/budget/adaptive) are documented in a reference doc and selectable via `model_profile` in config.json
  2. A project can set `models.gsd-planner: "claude-opus-4-5"` in config.json and the SDK routes that agent to the specified model
  3. `config.cjs` VALID_CONFIG_KEYS accepts all new fields without "unknown config key" warnings
  4. An existing project with only `granularity: "standard"` in config.json continues to work identically — no regressions
  5. Execution context profiles (dev/research/review) exist as reference docs that agents can reference to adjust behavior
**Plans**: 3 plans
Plans:
- [x] 02-01-PLAN.md — Extend config.cjs and core.cjs with new config fields (models.*, features.*, depth, context_window_tokens, execution_context)
- [x] 02-02-PLAN.md — Add adaptive model profile, update reference docs and settings UI
- [x] 02-03-PLAN.md — Create execution context profile files (dev, research, review)

### Phase 3: Gates & Execution Hardening
**Goal**: The gate system is formalized with a reusable gate-runner, plan-phase has stall-detection in its revision loop, wave execution is feature-flagged and manifest-tracked, and parallel executors handle git lock contention safely
**Depends on**: Phase 2
**Requirements**: GATES-01, GATES-02, GATES-03, GATES-04, GATES-05, GATES-06, GATES-07, GATES-08, GATES-09
**Success Criteria** (what must be TRUE):
  1. A gate reference doc exists documenting gate types (pre-flight, revision, abort, escalation) and the conditions that fire each
  2. Plan-phase workflow terminates with an escalation after 3 revision iterations rather than looping indefinitely
  3. Wave numbers are derived from `depends_on` frontmatter in PLAN.md files — gsd-planner writes them, execute-phase reads them
  4. A `.phase-manifest.json` is written to `.planning/` at phase completion containing the commit log for that phase
  5. Wave execution only activates when `workflow.wave_execution: true` is set in config.json — default is false and existing behavior is preserved
**Plans**: 4 plans
Plans:
- [x] 03-01-PLAN.md — Create gates.md and wave-execution.md reference docs
- [x] 03-02-PLAN.md — Harden plan-phase.md step 12 revision loop with stall detection
- [x] 03-03-PLAN.md — Update execute-phase.md (manifest write, git retry) and gsd-planner.md (@-references)
- [x] 03-04-PLAN.md — Add workflow.wave_execution feature flag to config.cjs, config.ts, and phase-runner.ts

### Phase 4: New User Skills
**Goal**: Four new user-facing commands exist and work end-to-end — explore for Socratic ideation, continue for headless next-step automation, undo for safe git revert, and import for external plan ingestion
**Depends on**: Phase 3
**Requirements**: SKILL-01, SKILL-02, SKILL-03, SKILL-04, SKILL-05, SKILL-06, SKILL-07, SKILL-08, SKILL-09, SKILL-10
**Success Criteria** (what must be TRUE):
  1. `/gsd:explore` accepts a freeform idea and returns structured output routed to one of: todos, seeds, notes, requirements, phases, or research questions
  2. `/gsd:continue` reads STATE.md status and executes the next appropriate action — and hard-stops at milestone boundaries, verification gaps, and errors rather than continuing blindly
  3. `/gsd:undo` can revert to the commit boundary of a given phase or plan using the phase manifest, after checking that no downstream phases depend on the target
  4. `/gsd:import` ingests an external PRD or plan file and surfaces BLOCKER/WARNING/INFO conflicts against existing project decisions before writing anything
  5. All four commands use `gsd-` prefixed agents and `/gsd:` command names — no PBR naming appears anywhere
**Plans**: 4 plans
Plans:
- [x] 04-01-PLAN.md — Create /gsd:explore command + workflow (Socratic ideation + output routing)
- [x] 04-02-PLAN.md — Create /gsd:continue command + workflow (state routing + hard stops)
- [x] 04-03-PLAN.md — Create /gsd:undo command + workflow (three-mode git revert + dependency check)
- [x] 04-04-PLAN.md — Create /gsd:import command + workflow (plan import + PRD extraction + conflict detection)

### Phase 5: Knowledge & Quality Uplift
**Goal**: GSD agents have structured thinking model guidance at decision points, the learnings system captures and feeds back per-phase insights, the scan command gives rapid codebase assessment, and status lines appear at workflow transitions
**Depends on**: Phase 1
**Requirements**: KNOWL-01, KNOWL-02, KNOWL-03, KNOWL-04, KNOWL-05, KNOWL-06, KNOWL-07
**Success Criteria** (what must be TRUE):
  1. Five thinking model reference docs exist (planning, execution, debug, research, verification) and at least one GSD agent references each at a relevant decision point
  2. `/gsd:scan` produces a rapid codebase assessment without invoking the full map-codebase workflow
  3. A LEARNINGS.md is written to `.planning/phases/{N}/` at phase completion — and gsd-planner reads the prior phase's LEARNINGS.md before producing a new plan
  4. Few-shot calibration examples exist for plan-checker and verifier agents, demonstrable by running them against a known-good and known-bad plan
  5. Status lines render phase name, current model profile, and context budget tier at workflow transition points
**Plans**: 4 plans
Plans:
- [ ] 05-01-PLAN.md — Create 5 thinking model reference docs + 2 few-shot calibration example files
- [x] 05-02-PLAN.md — Wire thinking model and few-shot @-refs into 6 GSD agents

### Phase 6: Intel System
**Goal**: A queryable codebase intelligence layer exists as a persistent JSON index under `.planning/intel/`, accessible via `/gsd:intel` command and `gsd-tools intel query` CLI subcommand, and gated behind a config flag
**Depends on**: Phase 1
**Requirements**: INTEL-01, INTEL-02, INTEL-03, INTEL-04, INTEL-05
**Success Criteria** (what must be TRUE):
  1. `/gsd:intel query <term>` returns matching results from `.planning/intel/` JSON files without re-scanning the codebase
  2. The intel index covers five artifacts: files.json, apis.json, deps.json, stack.json, and arch.md
  3. `gsd-tools intel query <term>` works as a CLI subcommand callable from workflows
  4. Intel system is inactive by default — setting `intel.enabled: true` in config.json activates it; omitting the key leaves existing behavior unchanged
**Plans**: 3 plans
Plans:
- [x] 06-01-PLAN.md — Create lib/intel.cjs (ported from PBR) and add intel.enabled to VALID_CONFIG_KEYS
